================================================================================
KEY FINDINGS
================================================================================

1. DISCOVERY WORKS BEST WITH HYBRID APPROACH
   - Keyword search (PostgreSQL tsvector) for recall
   - Semantic search (Vector DB) for relevance
   - Flat tag taxonomy (avoid hierarchies)
   - Reranking: keyword(0.3) + semantic(0.5) + popularity(0.2)
   Pattern: Combined approach beats any single method

2. VERSIONING MUST BE EXPLICIT (SemVer)
   - Standard: MAJOR.MINOR.PATCH[-prerelease][+build]
   - Constraint syntax: ^1.2.3, ~1.2.3, ranges
   - Prerelease lifecycle: alpha → beta → rc → release
   - Version resolution: DAG traversal for dependencies
   Pattern: npm's approach is battle-tested

3. METADATA MUST SUPPORT MULTIPLE DISCOVERY MODES
   - Git-centric (HF style): Single repo = source of truth
   - Manifest-centric (npm style): Separate skill.yaml file
   - Category-first: Primary category + unlimited tags
   - Rich documentation: README, examples, schema, API docs
   Pattern: skill.yaml includes everything needed for discovery

4. POSTGRESQL SCALES LARGE REGISTRIES
   - Core: skills, skill_versions, skill_tags tables
   - Search: tsvector index (GIN) for weighted full-text
   - Ranking: ts_rank() function + BM25 extensions
   - Constraints: Foreign keys + unique constraints
   Pattern: Append-only versions; update metadata via ON CONFLICT

5. CATEGORIES SHOULD BE FLAT & EXHAUSTIVE
   - 8 Primary categories (data, analytics, ai/ml, etc)
   - Unlimited flat tags (language, domain, complexity, status)
   - Discovery via tag filtering + keyword search
   Pattern: Taxonomy is governance tool, not technical constraint

================================================================================
POSTGRESQL SCHEMA (CORE)
================================================================================

CREATE TABLE skills (
  id BIGSERIAL PRIMARY KEY,
  name VARCHAR(255) UNIQUE NOT NULL,
  category VARCHAR(100) NOT NULL,
  description TEXT,
  search_vector tsvector  -- Full-text search
);

CREATE TABLE skill_versions (
  id BIGSERIAL PRIMARY KEY,
  skill_id BIGINT REFERENCES skills(id),
  version_semver VARCHAR(50) UNIQUE,
  is_prerelease BOOLEAN,
  requires TEXT  -- JSON: ["python>=3.11", "skill@^1.0.0"]
);

CREATE TABLE skill_tags (
  skill_id BIGINT REFERENCES skills(id),
  tag_name VARCHAR(100) NOT NULL,
  PRIMARY KEY (skill_id, tag_name)
);

CREATE INDEX skills_search_idx ON skills USING GIN(search_vector);

API SPECIFICATION (MVP)
================================================================================

POST /api/v1/skills/publish
  Input: { manifest: skill.yaml }
  Output: { id, version_id, published_at }

GET /api/v1/skills/search?query=X&category=Y&tags=Z&limit=20
  Output: [{ id, name, version, description, relevance_score }]

GET /api/v1/skills/{name}/resolve-version?constraint=^1.0.0
  Output: { resolved_version, available_versions, requires }

GET /api/v1/skills/{name}/versions
  Output: [{ version_semver, published_at, is_prerelease }]

DESIGN DECISIONS (Why)
================================================================================

Decision: PostgreSQL tsvector for search
Rationale: Native FTS; no extra infrastructure; GIN indexes scale
Trade-off: Must maintain search_vector via trigger/update

Decision: SemVer MAJOR.MINOR.PATCH
Rationale: Industry standard; clear compatibility signals
Trade-off: Complexity in constraint parsing

Decision: Flat tag taxonomy (not hierarchical)
Rationale: Flexible; avoids brittleness
Trade-off: Requires governance to prevent tag explosion

Decision: skill.yaml as manifest format
Rationale: Git-friendly; human-readable; GitHub/HF precedent
Trade-off: Schema evolution requires migration scripts

Decision: Hybrid search (keyword + semantic)
Rationale: Best of both worlds (recall + relevance)
Trade-off: Must maintain two indexes; complex ranking

Decision: Git repo as source of truth
Rationale: Immutable history; branching for versions
Trade-off: Requires version ↔ branch/tag mapping
